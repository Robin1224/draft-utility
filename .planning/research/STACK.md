# Stack Research

**Focus:** Real-time layer for a SvelteKit + PostgreSQL pick/ban draft (lobby presence, turn progression, team chat, spectator chat).  
**Researched:** 2026-04-03  
**Sources:** [svelte-realtime README](https://github.com/lanteanio/svelte-realtime) (HIGH), [npm `svelte-adapter-uws](https://www.npmjs.com/package/svelte-adapter-uws)` (HIGH), project `package.json` / `svelte.config.js` (HIGH), ecosystem search cross-check (MEDIUM).

## What NOT to Use


| Approach                                           | Why not (for this product)                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| -------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Polling as the primary sync**                    | Turn timer (default **30s**), presence, and chat need **sub-second** updates; polling wastes bandwidth and adds visible lag or race conditions. At most use polling as a **degraded fallback** if WebSockets are blocked (rare).                                                                                                                                                                                                                                 |
| **SSE-only for the full draft**                    | SSE is **server → client**. Every pick/ban would need a parallel path (HTTP POST/remote function), complicating ordering, retries, and connection auth. Fine as an **auxiliary** channel (e.g. read-only fanout) if you already standardize on WS for control plane.                                                                                                                                                                                             |
| **Liveblocks as the core draft engine**            | Strong for **CRDT-style collaborative documents** and hosted presence; **pricing and model** target collaborative SaaS, not a **rules-heavy game-style draft** with host authority, team-boundary chat, and spectator isolation. You would still encode most domain logic yourself; avoid unless you explicitly want their storage/presence product. **Confidence: MEDIUM** (product fit; Liveblocks remains viable for a narrow “presence UI” experiment only). |
| `**@sveltejs/adapter-node` + ad-hoc `ws` bolt-on** | Fighting the default adapter’s HTTP-centric model without the **Vite + hooks.ws + RPC** integration `svelte-realtime` already provides—you already depend on the latter two packages; **either commit to the uWS adapter stack or choose an external realtime host**, don’t half-integrate.                                                                                                                                                                      |
| **Ignoring DB persistence**                        | In-memory-only room state **will** diverge on deploy/restart and under multi-instance scaling unless you add **shared memory** (Redis) or **external party** (PartyKit, etc.). **PostgreSQL** should remain **source of truth** for completed drafts and critical room facts; the WS layer is for **live projection + fast path**.                                                                                                                               |


## Key Findings

- **Best default for this codebase:** **WebSockets via `svelte-realtime` + `svelte-adapter-uws`**, with `**hooks.ws.js` tied to Better Auth** and `**src/live/`** implementing room-scoped RPC and `publish` topics for team vs spectator chat and draft phase transitions—because those packages are **already dependencies** and the upstream README explicitly covers **streams, RPC, and cookie-based `upgrade`** (aligned with lobby/auth requirements in `PROJECT.md`).
- **Non-negotiable stack change:** Production today uses `**@sveltejs/adapter-node`** in `svelte.config.js` while realtime packages are unused; adopting this stack means **switching adapters** and **production hosting** that tolerates **uWebSockets.js** (native add-on). Plan a phase for **deploy/preview verification**, not only feature work.
- **Partykit / Liveblocks / SSE / polling** are **secondary or niche**: use **PartyKit** if hosting constraints rule out a single Node+uWS process; avoid **Liveblocks** as the main draft backbone; reserve **SSE/polling** for edge cases or degradation—not the happy path for pick/ban and chat.
